# SNCP 功能拆解

更新时间：2026-04-28

本文档回答三个问题：

1. 项目从用户视角到底在做什么。
2. 前端有哪些页面，后端有哪些模块。
3. 哪些业务规则是新人最容易忽略、但实际上会影响结果的。

## 1. 产品定位

`SNCP` 的核心目标不是做一个“泛用菜谱 App”，而是帮助用户围绕“吃了什么、合不合理、接下来怎么吃”建立一条完整闭环，重点服务中老年人和慢病饮食管理场景。

## 2. 核心业务闭环

当前实现的主链路可以概括为：

```text
注册 / 登录
  -> 完善健康档案与营养目标
  -> 记录一餐（手动录入 / AI 识图）
  -> 服务端汇总营养数据
  -> 今日看板给出评分、风险和建议
  -> 食谱推荐给出下一餐候选
  -> 长期趋势帮助复盘
```

管理端在这条链路之外，主要负责两件事：

- 维护公共健康禁忌规则
- 维护公共食谱库与后台用户管理

## 4. 后端模块拆解

| 模块 | 路由前缀 | 责任 |
|---|---|---|
| 认证 | `/api/auth` | 登录、注册、刷新令牌、当前用户信息、密码修改 |
| 档案 | `/api/profile` | 健康档案与营养目标 |
| 记餐 | `/api/meals` | 餐次创建、按天/区间查询、删除 |
| 看板 | `/api/dashboard` | 今日汇总、趋势统计 |
| 食谱 | `/api/recipes` | 个人食谱、公共食谱、内置食谱合并查询 |
| 提醒 | `/api/reminders` | 提醒的增删改查 |
| 管理 | `/api/admin` | 后台概览、用户列表、角色管理、健康规则 |
| AI | `/api/ai` | 识图、营养分析、推荐、食谱草稿抽取 |

## 5. 关键业务语义

### 5.1 角色不是只有“登录/未登录”

当前角色有三层：

- 普通用户：登录即可访问绝大多数功能
- `admin`：可访问管理后台、公共食谱和健康规则管理
- `webmaster`：在 `admin` 基础上，还可以给其他用户授予或撤销管理员权限

这套角色判断不是前端做的，而是后端在 `auth.py` 的装饰器里做的。

### 5.2 食谱库其实分三层

前端看到的“食谱”并不是一张表的简单查询，而是三类来源合并后的结果：

- 内置默认食谱：定义在 `backend/data/default_recipes.py`
- 服务端公共食谱：`recipes.user_id IS NULL`
- 用户私有食谱：`recipes.user_id = 当前用户`

同时接口还支持 `scope=all|local|server` 三种视图。

### 5.3 时间相关逻辑是“按用户本地日期”算的

后端把餐次时间统一存成 UTC，但看板和趋势按用户本地日期聚合。前端每次请求都会带：

- `X-Timezone-Offset-Minutes`
- `X-Client-Timezone`

如果这两个头丢了，接口还能工作，但统计结果可能跨天。

### 5.4 AI 不是单一路径

AI 能力的调用顺序不是固定绑死某一个供应商，而是按能力和配置逐层回退：

- 优先走独立 `AI_*` 远端接口（如果配置了）
- 否则优先用智谱
- 再退到本地规则
- 再退到本地规则/提示词兜底

所以开发时不能只测“有模型”的情况，也要测“没有模型配置”时页面是否能继续用。

### 5.5 看板和推荐带缓存，不是每次都重算

今日看板和食谱推荐都会把结果缓存到 Redis，并通过“用户状态版本号”决定缓存是否失效。以下数据变化会触发版本递增：

- 记餐新增/删除
- 健康档案修改
- 营养目标修改
- 食谱库修改

如果你改了业务逻辑却发现页面结果没变，先怀疑是不是缓存命中了。

## 6. 代码层职责边界

当前仓库的职责分层比较清晰：

- `SNCP-app/app`：页面与交互
- `SNCP-app/services`：前端 API 调用
- `SNCP-app/types`：前端结构定义
- `backend/routes`：接口入口与参数校验
- `backend/repository`：SQL 访问
- `backend/services`：认证、缓存、营养计算、文件处理等领域逻辑
- `ai_end`：AI 相关实现、技能和提示词

新人开发时建议尽量沿着这条边界扩展，不要把 SQL 写回前端，也不要把复杂业务判断塞进页面组件。

## 7. 信息来源

- 前端路由：`SNCP-app/app/_layout.tsx`、`SNCP-app/app/(tabs)/_layout.tsx`
- 服务调用：`SNCP-app/services/*.ts`
- 后端模块：`backend/app.py`、`backend/routes/*.py`
- AI 策略：`ai_end/services/ai_service.py`
